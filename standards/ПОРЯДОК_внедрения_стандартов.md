# Последовательность применения стандартов и системы worklog

Документ описывает порядок внедрения всех документов проекта — от чтения стандартов до сборки README. Каждый следующий шаг опирается на результаты предыдущего. Нарушение порядка ведёт к повторной работе.

---

## Документы проекта

### Группа B: Стандарты (governance)

Задают правила. Не меняются при старте нового проекта — только читаются и принимаются.

| Документ | Уровень | Область |
|----------|---------|---------|
| No-Unicode_Policy_v2.1.md | [C] + [W] + [I] | UI, продакшн-код, AI-чат, прототипы |
| MARKDOWN_STANDARD_RU_v2.1.md | [W] | README, документация проекта |
| MARKDOWN_STANDARD_EN_v2.1.md | [W] | То же на английском |
| README_TEMPLATE.md | - | Структура README.md |

### Группа A: Операционная система (worklog)

Инструменты работы. Копируются в проект и используются с первого дня.

| Документ | Назначение |
|----------|------------|
| README_WORKLOG.md | Руководство по системе worklog |
| TASK_TEMPLATE.md | Шаблоны промптов для подагентов |
| WORKLOG.md | Журнал работы агентов (живой файл) |

### Связь между группами

- Группа A подчиняется группе B (все .md файлы должны соответствовать MARKDOWN_STANDARD и No-Unicode Policy)
- Группа B не знает о группе A (стандарты не упоминают worklog)
- Группа A ссылается на группу B (README_WORKLOG упоминает стандарты)

## Полная последовательность (6 шагов)

### Шаг 1: Принять стандарты (группа B)

**Что делается:**

- Скопировать все файлы группы B в папку проекта (например, `docs/standards/`)
- Прочитать каждый стандарт полностью
- Убедиться, что команда понимает уровни [C], [W], [I]
- Определить подпись стека для проекта (формат: `Built with: <технологии>`, дефолт: см. README_TEMPLATE)

**Почему первым:** Стандарты — это фундамент. Всё последующее должно им соответствовать. Если начать работу без понимания правил, придётся переделывать.

**Риск:** Без этого шага каждый участник проекта будет оформлять код и документацию по-своему.

---

### Шаг 2: Развернуть систему worklog (группа A)

**Что делается:**

- Скопировать README_WORKLOG.md в проект (руководство)
- Скопировать TASK_TEMPLATE.md в проект (шаблоны промптов)
- Создать WORKLOG.md из шаблона (пустой журнал с заголовком)
- Проверить все три файла на соответствие MARKDOWN_STANDARD (подпись стека, языки у code blocks, маркеры списков)

**Почему вторым:** Worklog начинает фиксировать работу с шага 3. Без него — нет истории, нет координации между агентами, нет возможности откатиться.

**Почему после стандартов:** Файлы группы A должны соответствовать стандартам группы B с момента появления в проекте. Проверка на шаге 2 гарантирует, что worklog-система не нарушает MARKDOWN_STANDARD.

**Риск:** Если развернуть worklog до принятия стандартов — файлы могут не соответствовать правилам, и на шаге 5 их придётся переделывать.

---

### Шаг 3: REPRODUCIBILITY-STANDARD (Фундамент)

**Что делается:**

- Создаётся .env.example с относительными путями
- Настраивается db.ts: connection_limit=1, mkdirSync, относительные пути через process.cwd()
- Убираются абсолютные пути (/home/, http://localhost:)
- Проверяются зависимости (нет мёртвых пакетов)
- Настраивается dark theme через CSS-переменные
- Записать результат в WORKLOG.md (Task ID: 3)

**Почему третьим:** Инфраструктура должна быть стабильной до того, как код и документация начнут на неё опираться.

**Риск:** Если сначала написать UI, а потом исправить пути к БД или переменные окружения — придётся перетестировать весь код, который использовал эти подключения.

---

### Шаг 4: No-Unicode Policy v2.1 [C] (Защита UI-кода)

**Что делается:**

- Добавляется кастомное ESLint-правило no-unicode-policy/no-unicode (error)
- Заменяются все emoji в UI-компонентах на Lucide SVG-иконки
- Заменяются unicode-статусы на текстовые теги: [OK], [FAIL], [TODO]
- Запускается bun run lint для проверки
- Записать результат в WORKLOG.md (Task ID: 4)

**Почему четвёртым:** После настройки инфраструктуры (шаг 3) нужно «заморозить» код от unicode-загрязнений. ESLint-правило блокирует все новые emoji/unicode в коде. Файлы .ts/.tsx уже стабильны (шаг 3 прошёл), повторная очистка не потребуется.

**Риск:** Если сделать No-Unicode до Reproducibility — файлы .ts/.tsx будут изменены при настройке БД, и очистку придётся повторять. Если сделать после Markdown Standard — в .md останутся emoji, которые No-Unicode не ловит.

---

### Шаг 5: MARKDOWN_STANDARD v2.1 [W] (Документация)

**Что делается:**

- Удаляются emoji из всех .md файлов (worklog, README, AGENT_RULES и т.д.)
- Заменяется unicode-псевдографика на ASCII
- Проверяются списки (только `-`, не `*` или `+`)
- Проверяются code blocks (всегда с указанием языка, fallback: `text` или `bash`)
- Добавляется Stack Signature в корневые файлы (формат: `Built with: <технологии>`)
- Валидируются файлы группы A (README_WORKLOG.md, TASK_TEMPLATE.md, WORKLOG.md)
- Записать результат в WORKLOG.md (Task ID: 5)

**Почему пятым:** No-Unicode уже очистил код (шаг 4), но ESLint-правило не покрывает .md файлы. Markdown Standard — финальная зачистка для документации, включая файлы worklog-системы.

**Риск:** Если сделать Markdown Standard до No-Unicode — придётся повторно проходить по .md файлам, потому что оба стандарта запрещают emoji, но с разным охватом.

---

### Шаг 6: README_TEMPLATE (Финальная сборка)

**Что делается:**

- README.md собирается по шаблону (Title, Features, Tech Stack, Getting Started, Configuration, Project Structure, API Reference, Scripts, Development Rules, Agent Rules)
- Stack Signature добавляется в конец (конкретное значение стека проекта)
- Форматирование соответствует MARKDOWN_STANDARD
- Записать результат в WORKLOG.md (Task ID: 6)

**Почему последним:** README.md должен соответствовать MARKDOWN_STANDARD, поэтому шаблон применяется после того, как правила форматирования установлены (шаг 5). Технический стек уже определён на шаге 1, инфраструктура стабильна (шаг 3), код чист (шаг 4).

**Риск:** Если сделать README первым — при применении Markdown Standard придётся переформатировать списки (`*` в `-`), добавлять language к code blocks, добавлять Stack Signature.

## Диаграмма зависимостей

```text
Шаг 1: Стандарты (группа B)         Прочитать, принять правила
        |
        v
Шаг 2: Worklog (группа A)           Развернуть, проверить на соответствие B
        |
        v
Шаг 3: REPRODUCIBILITY              Настроить инфраструктуру (env, db, paths)
        |                            Записать в WORKLOG
        v
Шаг 4: No-Unicode Policy [C]        ESLint-правило + очистка UI-кода
        |                            Записать в WORKLOG
        v
Шаг 5: MARKDOWN_STANDARD [W]        Очистка .md файлов (включая группу A)
        |                            Записать в WORKLOG
        v
Шаг 6: README_TEMPLATE              Сборка README по шаблону
                                     Записать в WORKLOG
```

## Что происходит при нарушении порядка

| Ошибка | Последствие |
|--------|-------------|
| Worklog (шаг 2) до стандартов (шаг 1) | Файлы группы A не соответствуют B — повторная проверка и редактирование на шаге 5 |
| No-Unicode до Reproducibility | Файлы .ts/.tsx меняются при настройке БД — повторная очистка |
| Markdown Standard до No-Unicode | Emoji в .md и в коде — двойная проходка по файлам |
| README Template до Markdown Standard | Формат README не соответствует стандарту — переформатирование |
| Любой стандарт без Reproducibility | Абсолютные пути, мёртвые пакеты — скрытые баги |
| Reproducibility после README | Абсолютные пути в .env и db.ts ломают деплой — переписывание конфигурации и переформатирование README |
| Пропуск шага 2 (worklog) | Нет истории работы, нет координации между агентами, невозможно откатиться |

## Ключевые правила

1. **Слои:** Каждый шаг — это слой. Нижний слой должен быть стабильным, прежде чем класть следующий. Иначе придётся перекладывать слои заново.

2. **Группа A подчиняется группе B:** Все файлы worklog-системы проверяются на соответствие стандартам. Это делается дважды: при развёртывании (шаг 2) и при применении MARKDOWN_STANDARD (шаг 5).

3. **Worklog пишет каждый шаг:** Начиная с шага 3, каждый шаг фиксируется в WORKLOG.md. Шаги 1-2 — ручные (worklog ещё не используется), шаги 3-6 — с записью.

4. **Подпись стека определяется на шаге 1:** Формат задаёт MARKDOWN_STANDARD, конкретное значение — README_TEMPLATE. При смене стека проекта обновляется только README_TEMPLATE.
