# РУКОВОДСТВО ПО ОФОРМЛЕНИЮ MARKDOWN (v2.1)

> Уровень: **[W] Warning** | Область: README, документация проекта
>
> Связанный стандарт: **No-Unicode Policy v2.1** (для кода и UI - уровни [C]/[W]/[I])

---

## 1. Введение и цели

Данный стандарт устанавливает правила оформления Markdown-документации для обеспечения визуальной консистентности и профессионального уровня продукта.

**Цели:**

- Обеспечить визуальную согласованность документации
- Поддержать профессиональный уровень продукта
- Гарантировать контроль через дизайн-систему
- Исключить неконтролируемые визуальные артефакты

## 3. Запрещённые элементы

Использование следующих элементов **запрещено** в Markdown-файлах:

| Категория | Примеры | Примечание |
| :--- | :--- | :--- |
| **Эмодзи** | Любые пиктограммы: эмоции, объекты, UI-символы | Без исключений |
| **Unicode-иконки** | Статусы, действия, уведомления | Используйте текстовые теги |
| **Псевдографика таблиц** | Линии рамок (Unicode box-drawing) | Используйте MD-синтаксис |
| **Типографика в коде/заголовках** | Длинное тире `—` (ref), градус `°` (ref), копирайт `©` (ref) | Разрешены в обычном тексте, строго запрещены здесь |

**Охват запрета (для Типографики):**

- Заголовки и подзаголовки
- Таблицы (кроме справочных - см. ниже)
- Инлайн-код и блоки кода (комментарии, строки)
- Названия файлов и папок

**Исключение `(ref)` для справочных таблиц и code blocks:** Если цель ячейки таблицы или строки в code block - идентифицировать конкретный символ (чтобы показать, что именно запрещено или разрешено), допускается его указание с пометкой `(ref)`. Это не нарушает дух стандарта: символ используется как объект описания, а не как оформление. Без самого символа пример теряет смысл: «Неправильно: `—` (ref) в заголовке» - наглядно, «Неправильно: длинное тире в заголовке» - слепо.

## 4. Разрешённые элементы

### 4.1. Текстовые правила

Разрешено использовать:

| Категория | Примеры |
|-----------|---------|
| **ASCII буквы** | a-z, A-Z |
| **Буквы Unicode** | Языковые: кириллица (а-я, А-Я) и др. |
| **Цифры** | 0-9 |
| **Базовая пунктуация** | . , ; : ! ? - _ ( ) [ ] { } |
| **Типографские символы (только текст)** | `—` длинное тире (ref), `–` короткое тире (ref), `°` градус (ref), `©` копирайт (ref), `±` плюс-минус (ref) |
| **Пробелы** | space, tab, newline |

### 4.2. Whitelist для ASCII-диаграмм

Для технических диаграмм в документации **разрешены** следующие символы:

| Символ | Применение |
|--------|------------|
| -> | стрелка вправо |
| <- | стрелка влево |
| => | импликация |
| <= | обратная импликация |
| \| | вертикальная линия |
| + | соединение линий |
| - | горизонтальная линия |
| v | стрелка вниз |
| ^ | стрелка вверх |
| > | указатель |
| < | обратный указатель |

### 4.3. Синтаксис Markdown

| Элемент | Синтаксис |
|---------|-----------|
| Заголовки | `#`, `##`, `###` |
| Жирный | `**текст**` |
| Курсив | `*текст*` |
| Код инлайн | `` `код` `` |
| Блок кода | ` ```язык ` |
| Цитата | `>` |
| Ненумерованный список | `-` (строго) |
| Нумерованный список | `1.` |
| Ссылка | `[текст](url)` |
| Изображение | `![alt](url)` |

### 4.4. Текстовые теги для статусов

Вместо Unicode-символов используйте текстовые метки:

| Правильно | Неправильно |
|-----------|-------------|
| `[OK]` | `✓` (ref) |
| `[FAIL]` | `✗` (ref) |
| `[DONE]` | `✅` (ref) |
| `[TODO]` | `📋` (ref) |
| `[WARNING]` | `⚠` (ref) |
| `[INFO]` | `ℹ` (ref) |

## 5. Правила форматирования

### 5.1. Заголовки

- Используйте `#` для H1, `##` для H2 и т.д.
- Не используйте закрывающие символы `#`
- Только один H1 на документ
- Не используйте типографские символы (например, длинное тире) в заголовках

```text
Правильно:    # Заголовок
Неправильно:  # Заголовок #
Неправильно:  # Заголовок — (ref) Подзаголовок
```

Символ `—` (ref) в примере выше - демонстрация запрещённого символа, пометка `(ref)` указывает на справочное использование.

### 5.2. Списки

**Ненумерованные:**

- Всегда используйте `-` как единственный маркер для ненумерованных списков.
- Не смешивайте с `*` или `+`.

```text
Правильно:    - Пункт 1
Неправильно:  * Пункт 1
Неправильно:  -> Пункт 1
```

**Нумерованные:**

```text
1. Первый пункт
2. Второй пункт
```

### 5.3. Выделение текста

| Формат | Синтаксис |
|--------|-----------|
| Жирный | `**текст**` |
| Курсив | `*текст*` |
| Зачеркнутый | `~~текст~~` |

### 5.4. Оформление кода

**Inline код** (внутри текста):

```markdown
Используйте функцию `processFile()` для обработки.
```

**Блок кода** (с указанием языка):

````markdown
```typescript
const config = {
  encoding: 'utf-8',
  strict: true
};
```
````

**Правило для неизвестных языков:**
Если точный язык программирования или формат не поддерживается рендерером или неизвестен, всегда указывайте `text` или `bash` вместо пустого блока.

```text
Правильно:    ```text
Неправильно:  ```
```

**Правила:**

| Элемент | Формат | Назначение |
|---------|--------|------------|
| Inline код | `` `код` `` | Функции, переменные, команды в тексте |
| Блок кода | `` ```язык `` | Многострочный код, примеры, конфиги |

**НЕ используйте кастомные цвета:**

- Markdown не поддерживает цвета нативно
- HTML-теги типа `<span style="color:red">` могут блокироваться на GitHub
- Подсветка синтаксиса применяется автоматически рендерером
- Цвет — ответственность темы (GitHub, VS Code), не документа

## 6. Графические элементы

### 6.1. Базовое правило

Любой визуальный символ в документации = **только SVG** или **текстовая альтернатива**.

**Правило вставки SVG:**
Для вставки SVG используйте стандартный Markdown-синтаксис изображений. Сырые HTML-теги `<svg>...</svg>` запрещены для предотвращения XSS и проблем с рендерингом.

```markdown
![Описание иконки](./path/to/icon.svg)
```

### 6.2. Библиотека иконок

## 7. Подпись стека (Stack Signature)

Корневые файлы документации должны содержать подпись стека в конце файла.

**Область применения:**
- `README.md` (корневой)
- `CHANGELOG.md` (корневой)
- *Опционально для вложенных файлов в `docs/`, но не обязательно.*

**Формат:**

```markdown
---
Built with: <технологии проекта>
```

Конкретный стек определяется проектом, а не стандартом. Пример для Next.js-проектов:

```markdown
---
Built with: Next.js 16 + TypeScript + Tailwind CSS
```

Дефолтное значение для проектов данного стека см. в `README_TEMPLATE.md`.

**Правила:**

- Размещение: конец файла
- Разделитель: три дефиса `---`
- Содержимое: буквы, цифры, знаки `+` и `:`
- Графика запрещена

## 8. Контроль и Enforcement

### 8.1. Уровень [W] Warning - политика блокировки

| Этап | Действие | Блокирует merge? |
|------|----------|------------------|
| Code Review | Комментарий с требованием исправить | **Нет** |
| CI Pipeline | Warning в логах | **Нет** |
| Повторное нарушение | Escalation к Tech Lead | **Возможно** |

**Правило:** Уровень [W] не блокирует merge автоматически. Автор PR может:

1. Исправить нарушения и получить approval
2. Обосновать исключение в комментариях
3. Получить approval при консенсусе reviewer

**Escalation:** При систематических нарушениях (3+ PR без исправлений) — блокировка merge до обсуждения с Tech Lead.

### 8.2. Обязательная проверка всех .md файлов

**Правило:** Любой создаваемый или добавляемый `.md` файл в проект **обязан** проходить проверку и редактирование по данному стандарту.

| Действие | Когда | Ответственный |
|----------|-------|---------------|
| Создание нового .md | До commit | Автор |
| Добавление внешнего .md | До merge | Reviewer |
| Копирование .md из другого проекта | До commit | Автор |

**Процесс:**

1. Проверка по чек-листу (см. раздел 11)
2. Исправление нарушений
3. Добавление подписи стека (если это корневой файл)
4. Проверка в Code Review

**Автоматизация:**

```yaml
# .github/workflows/md-lint.yml
name: Markdown Lint
on: [push, pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install eslint-plugin-markdown
      - run: npx eslint --plugin markdown '**/*.md' --rule 'no-irregular-whitespace: error'
```

### 8.3. Критерии отклонения PR

PR **рекомендуется отклонить** при наличии:

1. Эмодзи в документации
2. Unicode-графики без текстовой альтернативы
3. Отсутствующей подписи стека в корневых технических документах (`README.md`, `CHANGELOG.md`)
4. Типографских символов внутри блоков кода или заголовков

### 8.4. Линтинг - этапы применения

| Этап | Когда | Инструмент | Действие |
|------|-------|------------|----------|
| Pre-commit | Перед коммитом | husky + lint-staged | Предупреждение |
| CI | Push в ветку | eslint-plugin-markdown | Warning в логах |
| Pre-merge | Перед merge в main | GitHub Action | Отчёт в PR |

**Конфигурация lint-staged:**

```json
{
  "*.md": [
    "eslint --plugin markdown --rule 'no-irregular-whitespace: error'"
  ]
}
```

**Очистка от эмодзи (применяется в CI):**

```javascript
replace(/[\u{1F000}-\u{1FFFF}]|[\u{2600}-\u{27BF}]|[\u{FE00}-\u{FEFF}]|[\u{1F900}-\u{1F9FF}]|[\u{2702}-\u{27B0}]/gu, '')
```

**Финальная санитизация (сохраняет whitelist диаграмм):**

```javascript
// Примечание: 'v' не указана явно, так как уже входит в базовый диапазон ASCII \x20-\x7E
replace(/[^\x20-\x7E\u0400-\u04FF\-\>\<\=\|\+\^]/g, '')
```

## 11. Чек-лист перед merge

- [ ] Нет эмодзи в документации
- [ ] Нет Unicode-иконок
- [ ] Нет типографских символов (длинное тире, копирайт и т.д.) в коде и заголовках
- [ ] Индикаторы статуса — текстовые теги `[OK]`, `[FAIL]`
- [ ] Ненумерованные списки используют строго маркер `-`
- [ ] Неизвестные блоки кода используют заглушку `text` или `bash`
- [ ] Подпись стека присутствует в корневых файлах
- [ ] Форматирование соответствует стандарту
- [ ] Диаграммы используют whitelist-символы
