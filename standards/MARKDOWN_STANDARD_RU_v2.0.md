# РУКОВОДСТВО ПО ОФОРМЛЕНИЮ MARKDOWN (v2.0)

> Уровень: **[W] Warning** | Область: README, документация проекта
>
> Связанный стандарт: **No-Unicode Policy v2.0** (для кода и UI - уровни [C]/[I])

---

## 1. Введение и цели

Данный стандарт устанавливает правила оформления Markdown-документации для обеспечения визуальной консистентности и профессионального уровня продукта.

**Цели:**

- Обеспечить визуальную согласованность документации
- Поддержать профессиональный уровень продукта
- Гарантировать контроль через дизайн-систему
- Исключить неконтролируемые визуальные артефакты

## 3. Запрещённые элементы

Использование следующих элементов **запрещено** в Markdown-файлах:

| Категория | Примеры | Примечание |
| :--- | :--- | :--- |
| **Эмодзи** | Любые пиктограммы: эмоции, объекты, UI-символы | Без исключений |
| **Unicode-иконки** | Статусы, действия, уведомления | Используйте текстовые теги |
| **Псевдографика таблиц** | Линии рамок: `│`, `─`, `┌`, `┐` | Используйте MD-синтаксис |
| **Спецсимволы типографики** | Длинное тире (`—`), градусы (`°`), копирайт (`©`) | Используйте `-` и `(c)` |

**Охват запрета:**

- Заголовки и подзаголовки
- Таблицы
- Списки
- Текстовое содержимое
- Блоки кода (комментарии, строки)

## 4. Разрешённые элементы

### 4.1. Текстовые правила

Разрешено использовать:

| Категория | Примеры |
|-----------|---------|
| **ASCII буквы** | a-z, A-Z |
| **Кириллица** | а-я, А-Я |
| **Цифры** | 0-9 |
| **Базовая пунктуация** | . , ; : ! ? - _ ( ) [ ] { } |
| **Пробелы** | space, tab, newline |

### 4.2. Whitelist для ASCII-диаграмм

Для технических диаграмм в документации **разрешены** следующие символы:

| Символ | Применение |
|--------|------------|
| -> | стрелка вправо |
| <- | стрелка влево |
| => | импликация |
| <= | обратная импликация |
| \| | вертикальная линия |
| + | соединение линий |
| - | горизонтальная линия |
| v | стрелка вниз |
| ^ | стрелка вверх |
| > | указатель |
| < | обратный указатель |

### 4.3. Синтаксис Markdown

| Элемент | Синтаксис |
|---------|-----------|
| Заголовки | `#`, `##`, `###` |
| Жирный | `**текст**` |
| Курсив | `*текст*` |
| Код инлайн | `` `code` `` |
| Блок кода | ` ```language ` |
| Цитата | `>` |
| Список | `-` или `*` или `1.` |
| Ссылка | `[текст](url)` |
| Изображение | `![alt](url)` |

### 4.4. Текстовые теги для статусов

Вместо Unicode-символов используйте текстовые метки:

| Правильно | Неправильно |
|-----------|-------------|
| `[OK]` | `✓` |
| `[FAIL]` | `✗` |
| `[DONE]` | `✅` |
| `[TODO]` | `📋` |
| `[WARNING]` | `⚠` |
| `[INFO]` | `ℹ` |

## 5. Правила форматирования

### 5.1. Заголовки

- Используйте `#` для H1, `##` для H2 и т.д.
- Не используйте закрывающие символы `#`
- Только один H1 на документ

```
Правильно:    # Заголовок
Неправильно:  # Заголовок #
```

### 5.2. Списки

**Ненумерованные:**

```
Правильно:    - Пункт 1
Неправильно:  * Пункт 1 (если начато с -)
Неправильно:  -> Пункт 1
```

**Нумерованные:**

```
1. Первый пункт
2. Второй пункт
```

### 5.4. Оформление кода

**Inline код** (внутри текста):

```markdown
Используйте функцию `processFile()` для обработки.
```

**Блок кода** (с указанием языка):

```markdown
```typescript
const config = {
  encoding: 'utf-8',
  strict: true
};
```
```

**Правила:**

| Элемент | Формат | Назначение |
|---------|--------|------------|
| Inline код | `` `код` `` | Функции, переменные, команды в тексте |
| Блок кода | `` ```язык `` | Многострочный код, примеры, конфиги |

**НЕ используйте кастомные цвета:**

- Markdown не поддерживает цвета нативно
- HTML-теги типа `<span style="color:red">` могут блокироваться на GitHub
- Подсветка синтаксиса применяется автоматически рендерером
- Цвет — ответственность темы (GitHub, VS Code), не документа

## 7. Подпись стека (Stack Signature)

Каждый технический документ должен содержать подпись стека в конце файла.

**Формат:**

```markdown
---
Built with: Next.js 16 + TypeScript + Tailwind CSS
```

**Правила:**

- Размещение: конец файла
- Разделитель: три дефиса `---`
- Содержимое: латиница, кириллица, цифры, знаки `+` и `:`
- Графика запрещена

## 8. Контроль и Enforcement

### 8.1. Уровень [W] Warning — политика блокировки

| Этап | Действие | Блокирует merge? |
|------|----------|------------------|
| Code Review | Комментарий с требованием исправить | **Нет** |
| CI Pipeline | Warning в логах | **Нет** |
| Повторное нарушение | Escalation к Tech Lead | **Возможно** |

**Правило:** Уровень [W] не блокирует merge автоматически. Автор PR может:

1. Исправить нарушения и получить approval
2. Обосновать исключение в комментариях
3. Получить approval при консенсусе reviewer

**Escalation:** При систематических нарушениях (3+ PR без исправлений) — блокировка merge до обсуждения с Tech Lead.

### 8.2. Обязательная проверка всех .md файлов

**Правило:** Любой создаваемый или добавляемый `.md` файл в проект **обязан** проходить проверку и редактирование по данному стандарту.

| Действие | Когда | Ответственный |
|----------|-------|---------------|
| Создание нового .md | До commit | Автор |
| Добавление внешнего .md | До merge | Reviewer |
| Копирование .md из другого проекта | До commit | Автор |

**Процесс:**

1. Проверка по чек-листу (см. раздел 11)
2. Исправление нарушений
3. Добавление подписи стека
4. Проверка в Code Review

**Автоматизация:**

```yaml
# .github/workflows/md-lint.yml
name: Markdown Lint
on: [push, pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npx eslint '**/*.md' --rule 'no-irregular-whitespace: error'
```

### 8.3. Критерии отклонения PR

PR **рекомендуется отклонить** при наличии:

1. Эмодзи в документации
2. Unicode-графики без текстовой альтернативы
3. Отсутствующей подписи стека (для технических документов)

### 8.4. Линтинг — этапы применения

| Этап | Когда | Инструмент | Действие |
|------|-------|------------|----------|
| Pre-commit | Перед коммитом | husky + lint-staged | Предупреждение |
| CI | Push в ветку | eslint-plugin-markdown | Warning в логах |
| Pre-merge | Перед merge в main | GitHub Action | Отчёт в PR |

**Конфигурация lint-staged:**

```json
{
  "*.md": [
    "eslint --rule 'no-irregular-whitespace: error'"
  ]
}
```

**Очистка от эмодзи (применяется в CI):**

```javascript
replace(/[\u{1F000}-\u{1FFFF}]|[\u{2600}-\u{27BF}]|[\u{FE00}-\u{FEFF}]|[\u{1F900}-\u{1F9FF}]|[\u{2702}-\u{27B0}]/gu, '')
```

**Финальная санитизация (сохраняет whitelist диаграмм):**

```javascript
replace(/[^\x20-\x7E\u0400-\u04FF\-\>\<\=\|\+\^v]/g, '')
```

## 11. Чек-лист перед merge

- [ ] Нет эмодзи в документации
- [ ] Нет Unicode-иконок
- [ ] Индикаторы статуса — текстовые теги `[OK]`, `[FAIL]`
- [ ] Подпись стека присутствует
- [ ] Форматирование соответствует стандарту
- [ ] Диаграммы используют whitelist-символы
